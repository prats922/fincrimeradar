# FinCrimeRadar Backlog Board

Three loops, rotating. Each working session picks one loop, clears or advances what's in it, then moves to the next. Don't run more than one loop per session, that's how things end up half finished.

Suggested cadence: fortnightly rotation, Polish → Build → Content → repeat. Polish goes first because it's the smallest and the most likely to get skipped if it's not first.

- [ ] **Scenario Lab, SAR Writing Sandbox module** — practice module where users write a Suspicious Activity Report narrative against a case dossier and get structured feedback. Originated from a PRD draft, redesigned during scoping discussion: build as a new Scenario Lab module reusing existing dashboard/case-data/consent infrastructure, not a standalone three-panel product. Key architectural decision: separate LLM extraction from scoring. One Claude API call extracts structured facts only (which of the 5 Ws were addressed with supporting quotes, which dossier red flags were mentioned, any subjective/speculative language found verbatim, whether a specific transaction detail was cited), no scores or point deductions generated by the model itself. All actual scoring (fixed deductions per missed red flag, per subjective phrase, per structural gap) happens in deterministic code against the extracted facts, not inside the LLM's own arithmetic, since LLMs are unreliable at consistent multi-step scoring within a single generation. Editor built as three separate labelled text boxes (Intro, Investigative Body, Final Disposition), not one rich-text field, so structural completeness is a trivial check rather than requiring LLM judgement. Output is a "Practice Case Summary" PDF, explicitly framed as self-directed practice, not "verified" or "authenticated" (site is a personal project, not an institution). Has ongoing per-submission API cost, needs session-based rate limiting before launch. Phase 0 scope: one case, two panels, extraction plus deterministic scoring working end to end, no PDF yet, prove score stability (same narrative run through extraction 3x should produce consistent results) before building anything further. Competes with the guide chatbot for build time and API budget, sequence against that and against finishing Fraud/Risk Scoring first. Needs a full dedicated scoping session before any code, case dossier structure and the extraction prompt need to be designed properly, same accuracy discipline that caught the Horizon Shipping and UK CTR errors applies here, arguably with higher stakes since this actively grades a user's professional judgement.
- [ ] **Guide chatbot** — needs a full scoping session before build starts. RAG over Knowledge Hub content (MLRO Handbook, AML/PEP/SAR/FATF/Crypto guides), embedded on guide pages so visitors can ask questions answered strictly from FinCrimeRadar's own content, not a general model. Proposed shape: retrieval layer over chunked Knowledge Hub content, new endpoint on `fincrimeradar-api` (FastAPI/Render, reuses existing infra), generation via Claude API, hard constraint to only answer from retrieved content with a visible fallback ("not covered in our guides, consult a professional") when nothing relevant is retrieved.
